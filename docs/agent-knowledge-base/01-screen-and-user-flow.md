# 界面理解与交互参考

本知识库参考当前黑马点评界面风格：顶部城市与搜索框，首页类目入口，瀑布流探店内容，底部导航；类目页提供距离、人气、评分等筛选项。

## 首页用户意图

首页截图中用户能看到：

- 城市：杭州。
- 搜索框：请输入商户名、地点。
- 类目入口：美食、KTV、丽人美发、美睫美甲、按摩足疗、美容 SPA、亲子游乐、酒吧、密室、健身运动。
- 探店内容流：用户发布的图文笔记。
- 底部导航：首页、地图、发布、消息、我的。

Agent 可以在首页承担“缩短搜索路径”的角色。

建议开场按钮：

- 附近美食
- 查优惠券
- 帮我选类目
- 今天适合去哪
- 商家运营建议

## 类目页用户意图

美食类目页截图中能看到：

- 顶部标题：美食。
- 搜索入口。
- 筛选项：美食、距离、人气、评分。
- 店铺列表：店名、评分、评论数、人均、商圈、距离、地址、图片。

用户在这个页面最常见的问题：

- 哪家离我最近。
- 哪家评分高。
- 哪家人均便宜。
- 哪家评论多，比较稳。
- 这家有没有优惠券。
- 适合几个人去。
- 有没有适合约会、朋友聚会、亲子、夜宵的店。

## Agent 悬浮按钮设计

如果未来改前端，可以在右侧中部放一个小的 AI 悬浮按钮。按钮不要遮挡门店文字和底部导航。

按钮状态：

- 默认：AI 小图标。
- 有推荐：显示“帮你选”。
- 用户停留在店铺页：显示“查券”。
- 用户停留在类目页：显示“筛选”。

点击后出现半屏抽屉，不建议全屏覆盖，因为用户还要对照列表。

## 不改前端时的替代方案

不改 Nginx 前端时，可以用 Dify 发布的 WebApp 页面作为独立入口。

入口文案可以是：

> 不想慢慢筛？打开 AI 导购助手，一句话找店、查券、看规则。

在 README 或演示文档中放链接即可。也可以让用户复制店铺名到 Agent 中查询优惠券。

## 用户路径 1：附近店铺

用户：附近有什么好吃的？

Agent：

1. 判断是“附近店铺查询”。
2. 如果没有位置，询问是否使用默认城市或输入商圈。
3. 调用后端店铺接口。
4. 输出店铺卡片：
   - 店名。
   - 评分。
   - 人均。
   - 距离。
   - 地址。
   - 推荐理由。
5. 给按钮：
   - 查第 1 家优惠券。
   - 按评分重排。
   - 100 元以内。
   - 换一批。

## 用户路径 2：优惠券助手

用户：新白鹿有没有券？

Agent：

1. 判断是“优惠券查询”。
2. 如果只有店名，先调用店铺搜索拿到 shopId。
3. 调用优惠券列表接口。
4. 输出优惠券：
   - 券名。
   - 支付金额。
   - 抵扣金额。
   - 使用规则。
   - 是否普通券或秒杀券。
5. 给按钮：
   - 查看使用规则。
   - 领取普通券。
   - 秒杀券提醒。

## 用户路径 3：知识库问答

用户：第一次约会选餐厅应该看什么？

Agent：

1. 判断是“消费决策知识问答”。
2. 检索知识库，不调用后端。
3. 回答选择逻辑：
   - 优先看评分和评论数。
   - 人均与预算匹配。
   - 距离不要太远。
   - 环境关键词更重要。
   - 有券不等于最适合。
4. 给按钮：
   - 帮我找附近约会餐厅。
   - 100 元以内。
   - 安静环境。
   - 查优惠券。

## 页面文案原则

- 不要让用户理解技术词。
- 不说“我将调用 API”。
- 不说“我从数据库读取”。
- 面向用户说“我帮你查一下附近门店”。
- 面向商家说“我帮你整理一份活动说明”。
- 面向运营说“我给你一套客服回复口径”。
