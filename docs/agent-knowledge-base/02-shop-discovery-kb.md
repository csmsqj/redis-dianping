# 附近店铺与筛选知识库

本文件用于回答“怎么选店”“筛选项是什么意思”“为什么推荐这些指标”等非实时问题。具体门店列表、距离、评分、人均价格必须以后端查询结果为准。

## 用户找店的核心需求

用户找附近店铺，本质上通常不是只想看列表，而是想降低选择成本。常见需求包括：

- 离我近。
- 评分高。
- 人均符合预算。
- 评论数量足够多。
- 类目符合场景。
- 有优惠。
- 适合现在就去。
- 适合某种关系或场景，如朋友聚会、约会、亲子、独自一人。

Agent 回答时应先判断用户属于哪种决策模式。

## 四种常见决策模式

### 快速就近型

用户表达：

- 附近随便吃点。
- 离我最近的。
- 现在就想去。
- 不想排队。

推荐逻辑：

- 优先距离。
- 其次营业时间。
- 再看评分是否过低。
- 价格只做基础过滤。

回答模板：

> 你这个需求适合按距离优先，我会先找近的，再排除评分明显偏低或价格不合适的店。

### 品质稳妥型

用户表达：

- 推荐靠谱一点的。
- 不想踩雷。
- 请客吃饭。
- 评分高一点。

推荐逻辑：

- 评分优先。
- 评论数不能太少。
- 人均与预算匹配。
- 店铺图片和评价关键词辅助判断。

回答模板：

> 想稳一点，不能只看评分，还要看评论数。评分高但评论很少，可靠性不如评分略低但评论很多的店。

### 预算控制型

用户表达：

- 100 元以内。
- 便宜一点。
- 学生党。
- 性价比高。

推荐逻辑：

- 人均价格优先。
- 优惠券作为加分项。
- 距离不能太远，否则时间成本变高。
- 评分低于基本线时不推荐。

回答模板：

> 性价比不是只看低价，而是在人均、评分、距离和优惠之间找平衡。

### 场景匹配型

用户表达：

- 约会。
- 朋友聚餐。
- 带孩子。
- 公司团建。
- 夜宵。
- 拍照好看。

推荐逻辑：

- 先匹配类目和场景。
- 再筛评分、人均、距离。
- 最后看是否有优惠。

回答模板：

> 场景型选择要先看适配度。比如约会更看环境和安静程度，朋友聚餐更看菜品丰富度和座位容量。

## 筛选项解释

### 距离

距离适合解决“快到达”的问题。距离近不一定最好，但适合午餐、临时安排、下班后快速消费。

注意事项：

- 距离受定位准确度影响。
- 直线距离不等于步行距离。
- 商场内部店铺可能需要额外找楼层。

### 人气

人气通常反映浏览、销量、评论、收藏或近期热度。人气高说明选择的人多，但不一定完全适合当前用户。

适合：

- 不知道怎么选。
- 想找热门店。
- 第一次到某个商圈。

不适合：

- 追求安静。
- 不想排队。
- 预算很低。

### 评分

评分反映用户满意度，但要结合评论数量看。

判断方法：

- 评分高且评论多：稳妥。
- 评分高但评论少：需要谨慎。
- 评分一般但销量高：可能适合大众刚需。
- 评分低且评论多：不建议优先推荐。

### 人均

人均用于判断预算，不代表最终价格。实际消费会受菜品、人数、套餐、服务费、优惠券影响。

回答时不要承诺用户最终支付金额，只能说“大致预算参考”。

## 推荐理由生成规则

Agent 推荐店铺时，每家店给 1 到 2 条理由即可，避免冗长。

可用理由：

- 距离近，适合马上出发。
- 评分较高，适合稳妥选择。
- 评论量多，参考样本更充分。
- 人均符合预算。
- 商圈明确，方便到达。
- 有优惠券，可以进一步查券。
- 类目适合当前场景。

不要使用的理由：

- “绝对最好吃”。
- “一定不踩雷”。
- “保证有座位”。
- “保证券能抢到”。

## 无结果时的处理

如果附近店铺没有结果，不要简单回答“没有”。应该引导用户换条件：

- 扩大距离范围。
- 换一个商圈。
- 降低评分条件。
- 换相近类目。
- 改用关键词搜索。

回答模板：

> 当前条件下没有合适结果。可以先把范围放宽一点，或者换成相近类目，比如从“烤肉”放宽到“美食”，再按评分筛选。

## 适合做按钮的筛选项

- 距离最近。
- 评分最高。
- 人气最高。
- 人均 100 内。
- 有优惠券。
- 适合约会。
- 适合朋友聚会。
- 适合亲子。
- 换一批。

## 面向企业的价值

附近店铺查询 Agent 能减少用户在列表中反复筛选的时间，让用户更快进入“查看门店详情”和“查优惠券”阶段。对平台来说，这能提升从浏览到下单的转化；对商家来说，可以减少“你家在哪”“人均多少”“有没有优惠”这类重复咨询。
