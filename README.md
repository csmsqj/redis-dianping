# 黑马点评（Redis 进阶实战版）

## 一、项目简介

这是一个基于 **Spring Boot + MyBatis-Plus + MySQL + Redis** 实现的黑马点评实战项目。我在原有业务功能的基础上，重点把项目往 **“高并发场景下的数据一致性、缓存设计、登录态共享、社交互动、秒杀异步化”** 这几个方向做了完整落地。

项目不只是把接口跑通，而是尽量把真实后端开发里常见的几个难点串起来：

- 如何用 Redis 缓解数据库压力
- 如何处理缓存穿透、缓存击穿、缓存雪崩
- 如何把 Session 型登录改造成 Redis Token 登录
- 如何利用 Redis 多种数据结构做点赞、签到、共同关注、附近商铺、Feed 流
- 如何把秒杀下单从“同步查库写库”改造成“Redis 原子校验 + Stream 异步下单”
- 如何在高并发下保证“一人一单”、库存正确、异常消息可补偿处理

这个项目更偏向 **Redis 全场景综合训练**，既包含基础缓存能力，也包含分布式锁、Lua、Stream、Bitmap、GEO、ZSet、Set 等典型实战方案。

## 二、项目亮点

### 1. 缓存体系比较完整

项目里不是只做了“查一次 Redis，查不到再查数据库”这么简单，而是把缓存场景拆得比较细：

- **缓存穿透**：商铺查询查不到时缓存空值，避免恶意请求反复打数据库
- **缓存击穿**：热点商铺缓存重建时，使用自定义分布式锁控制并发重建
- **缓存雪崩**：为部分缓存 TTL 增加随机值，避免大量 key 同时过期
- **缓存更新**：店铺信息更新时采用“先更新数据库，再删除缓存”的方式降低脏数据风险

### 2. 登录态从 Session 转成 Redis 共享存储

项目把验证码、Token、用户登录信息都放到了 Redis 中，解决了传统 Session 在多实例部署下不共享的问题：

- 手机号验证码存 Redis
- 登录成功后生成 Token
- 用户信息转成 Hash 存 Redis
- 通过拦截器完成用户信息刷新与 ThreadLocal 保存
- 每次请求刷新 Token TTL，形成“滑动过期”体验

这样做更接近真实项目里的无状态登录方案，也更适合后续做集群扩展。

### 3. 秒杀链路做了 Redis 原子化 + 异步化

这是本项目最核心的高并发亮点之一。

秒杀请求不是直接进入数据库做“查库存、查订单、扣库存、写订单”这条慢链路，而是拆成两段：

- **第一段：Redis + Lua 原子校验**
    - 校验库存是否充足
    - 校验是否已经下过单
    - 原子扣减 Redis 库存
    - 原子记录一人一单
- **第二段：Redis Stream 异步下单**
    - 后台线程消费订单消息
    - 事务中执行数据库扣库存与创建订单
    - ACK 确认消息消费成功
    - 出现异常时处理 Pending List，补偿未确认消息

这条链路相当于把高并发热点流量先挡在 Redis 层，再把数据库写操作异步串行化处理，明显降低了数据库瞬时压力。

### 4. Redis 数据结构使用比较丰富

项目不是单一用 Redis 做缓存，而是把 Redis 多种数据结构都落到了具体业务里：

- **String**：验证码、商铺缓存、秒杀库存
- **Hash**：登录用户信息、秒杀时间信息
- **List**：商铺类型缓存
- **Set**：关注关系、共同关注、一人一单集合标记
- **ZSet**：博客点赞、粉丝收件箱 Feed 流
- **Bitmap**：用户签到、连续签到统计
- **GEO**：附近商铺查询
- **Stream**：秒杀异步下单消息队列

这类项目在面试里会比较有优势，因为能够直接证明你不是只会“背 Redis 八股”，而是真正把数据结构映射到了业务建模里。

## 五、Redis 在项目中的具体落地

### 1. 验证码与登录 Token

用户请求验证码时，将验证码存入 Redis；登录成功后生成随机 Token，把脱敏后的用户信息转成 Hash 存入 Redis。这样可以避免把完整用户对象放入 Session，也解决了多实例登录态共享问题。

### 2. 商铺缓存

商铺查询优先读 Redis：

- 命中则直接返回
- 未命中则尝试获取分布式锁
- 获取成功后查数据库并回写缓存
- 数据不存在时缓存空字符串，防止穿透
- 回写缓存时附带随机 TTL，缓解雪崩

### 3. 商铺类型缓存

商铺类型列表使用 Redis List 存储，减少高频读取数据库。

### 4. 附近商铺查询

将店铺地理位置存入 Redis GEO，通过坐标半径搜索拿到商铺 ID 与距离，再回表查询详情并按距离组装返回。

### 5. 点赞功能

博客点赞使用 ZSet 存储：

- member：用户 ID
- score：点赞时间戳

这样既能判断用户是否点过赞，也能按时间顺序查询点赞用户列表。

### 6. 关注与共同关注

关注关系除了落库，还会同步写入 Redis Set。这样在做“共同关注”时，就可以直接用 Set 的交集运算完成，不必多次查表再在 Java 里比对。

### 7. Feed 流收件箱

用户发布博客后，会把博客 ID 推送到粉丝对应的 ZSet 收件箱中：

- key：`blog:feed:{userId}`
- member：博客 ID
- score：发布时间戳

读取时使用按 score 倒序 + 滚动分页的方式，适合关注流场景。

### 8. Bitmap 签到

签到使用 Bitmap：

- 每月一个 key
- 每天用一个 bit 位表示是否签到
- 连续签到统计通过 `BITFIELD` 取出位串后再做位运算统计

这个方案空间占用小，也非常适合签到场景。

### 9. 秒杀库存与时间窗口

新增秒杀券时，项目会把：

- 库存写入 Redis String
- 秒杀开始时间 / 结束时间写入 Redis Hash

这样真正秒杀时，就可以先在 Redis 层做快速校验。

### 10. 秒杀异步队列

项目使用 Redis Stream 存储秒杀订单消息，后台线程持续消费；如果消息处理异常，程序会继续扫描 Pending List 补偿未 ACK 的旧消息。

## 六、项目中使用了什么锁

### 1. 自定义分布式锁：SimpleRedisLock

项目在商铺热点缓存重建时，使用了自定义 Redis 分布式锁：

- 基于 `SETNX + EXPIRE` 思路实现加锁
- 锁 value 由“JVM 唯一标识 + 线程 ID”组成，避免误删他人锁
- 解锁不是直接 `DEL`，而是通过 Lua 脚本先判断再删除，保证原子性

这个锁主要用于 **缓存击穿场景下的并发缓存重建控制**。

### 2. Lua 原子脚本

秒杀场景虽然不是传统意义上的“加锁”，但本质上通过 Lua 把多步操作打包成一个 Redis 原子操作，效果上相当于把“库存判断 + 一人一单判断 + 扣库存 + 记录购买状态”锁在一次执行里。

相比把这些逻辑拆到 Java 中分步执行，Lua 的优势是：

- 原子性更强
- 网络往返更少
- 更适合秒杀这种极高并发热点场景

### 3. 数据库层并发兜底

真正落库时，项目又加了一层数据库兜底：

- 通过 `stock > 0` 条件更新实现乐观扣减库存
- 通过 `DuplicateKeyException` 处理重复下单冲突

也就是说，这个项目的并发控制并不是“只靠某一种锁”，而是 **Redis 前置削峰 + Lua 原子校验 + 数据库乐观更新 / 唯一约束兜底** 的组合方案。

## 七、秒杀核心链路

### 1. 秒杀券写入阶段

新增秒杀券时：

1. 写数据库中的优惠券与秒杀券信息
2. 把库存写入 Redis
3. 把开始时间和结束时间写入 Redis

### 2. 用户下单阶段

用户请求秒杀时：

1. 先在 Java 中校验秒杀时间是否合法
2. 调用 Lua 脚本到 Redis 做原子判断
3. Lua 返回成功后，生成全局订单 ID
4. 将订单任务交给 Stream 异步处理
5. 接口快速返回，减少请求阻塞时间

### 3. 异步消费阶段

后台线程启动后会持续消费 `stream.orders`：

1. 读取订单消息
2. 转换成订单对象
3. 调用事务服务创建订单
4. 成功后 ACK
5. 失败则进入异常处理逻辑

### 4. Pending List 补偿阶段

如果消费者处理消息异常但未 ACK，消息会留在 Pending List 中。项目中专门实现了 `handlePendingList()`：

- 从 Pending List 读取历史未确认消息
- 继续执行创建订单逻辑
- 对重复订单进行异常识别并 ACK
- 尽量避免消息长期积压或订单丢失

这一块是很多初学项目容易遗漏的地方，而这个项目已经把它补上了。

## 九、如何启动项目

### 1. 环境准备

你需要先准备：

- JDK 21
- Maven
- MySQL
- Redis

### 2. 数据库准备

执行 `src/main/resources/db` 下的 SQL 脚本，初始化 `hmdp` 数据库。

### 3. 修改配置

当前项目使用：

- `application.yaml` 指定端口和激活环境
- `application-dev.yaml` 配置 MySQL 与 Redis 连接

启动前请根据自己的环境修改：

- MySQL 用户名 / 密码
- Redis 地址 / 端口 / 密码
- `RedissionConfig` 中的 Redis 单机地址

### 4. 运行项目

```bash
mvn clean package
mvn spring-boot:run
```

默认端口：

```text
8081
```
